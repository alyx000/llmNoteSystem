# AI 应该帮助我们产出更好的代码

## 中文翻译

### AI 应该帮助我们产出更好的代码

许多开发者担心，把代码外包给 AI 工具会导致质量下降：糟糕代码被高速产出，以至于决策者愿意忽略其中的问题。

如果采用 coding agent 明显降低了你产出的代码和功能质量，你应该直接处理这个问题：找出流程中哪些环节正在伤害输出质量，然后修复它们。

用 agent 交付更差的代码是一种选择。我们也可以选择交付更好的代码。

### 避免背上技术债

我喜欢用技术债的视角思考如何交付更好的代码。技术债来自权衡：用“正确方式”做太耗时，所以我们在时间约束下先凑合，并希望项目能活到以后有机会偿还债务。

缓解技术债最好的办法，是一开始就避免背上它。

根据我的经验，很多技术债修复都属于“概念上简单，但很耗时间”的改动。

- 早期 API 设计没有覆盖后来出现的重要场景。修正这个 API 需要改几十处代码，于是更快的做法是加一个略有差异的新 API，并接受重复。
- 早期给某个概念起了不好的名字，比如用了 teams 而不是 groups，但全代码库清理命名太费工，所以只在 UI 上修正。
- 系统随着时间增长，出现了重复但略有差异的功能，需要合并和重构。
- 某个文件膨胀到几千行，理想情况下应该拆成多个模块。

这些改动在概念上都很简单，但仍然需要投入时间；在更紧急的问题面前，这种投入很难被证明合理。

### Coding agent 可以替我们处理这些事

这类重构任务是 coding agent 的理想应用场景。

启动一个 agent，告诉它要改什么，然后让它在后台的分支或 worktree 里慢慢处理。

我通常会用异步 coding agent 做这类事，比如 Gemini Jules、OpenAI Codex web 或 Claude Code on the web。这样我可以运行这些重构任务，同时不打断自己在笔记本上的工作流。

然后在 Pull Request 里评估结果。好的就合并；差一点的，就继续提示它该怎么改；很差的，就丢掉。

这些代码改进的成本已经降得很低，所以我们可以对小的代码异味和不便采取零容忍态度。

### AI 工具让我们能考虑更多选项

任何软件开发任务都有大量可能的解决路径。很多重要技术债都来自规划阶段的错误选择：错过显而易见的简单方案，或选了后来证明并不完全合适的技术。

LLM 可以帮助我们避免漏掉那些原本没想到的明显方案。它们只会建议训练数据中常见的方案，但这些方案往往正是最可能有效的“无聊技术”。

更重要的是，coding agent 可以帮助做探索性原型。

做出有信心的技术选择，最好的方法是用原型证明它适合目标场景。

Redis 是否适合一个预期有数千并发用户的网站活动流？

最可靠的办法是搭一个系统模拟，跑负载测试，看看哪里会坏。

Coding agent 可以通过一个精心编写的 prompt 构建这种模拟，把实验成本降到几乎为零。正因为它们如此便宜，我们可以同时运行多个实验，测试几种方案，然后选择最适合当前问题的那一个。

### 拥抱 compound engineering 循环

Agent 会遵循指令。我们可以根据过去学到的东西，不断演化这些指令，让未来运行得到更好的结果。

Every 的 Dan Shipper 和 Kieran Klaassen 把他们公司使用 coding agent 的方法称为 Compound Engineering。每个 coding 项目结束时，他们都会做一次复盘，称为 compound step：把有效做法记录下来，供未来 agent 运行使用。

如果我们想从 agent 身上得到最好的结果，就应该持续提高代码库质量。小改进会复合增长。过去非常耗时的质量增强，现在成本已经低到没有借口不在交付新功能的同时投资质量。Coding agent 意味着我们终于可以两者兼得。

## 提炼

### 核心论点

Simon Willison 这篇文章的重点是：AI agent 不应该成为“更快地产出更差代码”的借口。相反，agent 应该降低重构、技术债清理、探索性原型和质量改进的成本，让团队在交付新功能的同时持续提升代码质量。

他把技术债拆成一类常见但容易被拖延的问题：命名不一致、API 设计补丁、重复功能、巨型文件拆分。这些问题通常不难理解，但人工处理耗时，所以常被排在更紧急需求之后。Coding agent 改变的是成本结构：以前“不值得专门花时间清理”的小债务，现在可以交给后台 agent 在独立分支里处理。

### 关键洞察

- 使用 agent 后代码变差，不是必然结果，而是流程选择。若质量下降，应修流程，而不是接受低质量。
- Agentic refactoring 的最佳场景，是概念简单但改动面广、机械工作多、人工时间不划算的任务。
- PR 仍然是质量闸门。Agent 可以后台跑，但结果要通过人类评审、继续提示修正或直接丢弃。
- AI 降低了探索性原型的成本。技术选型不必只靠讨论，可以让 agent 快速搭模拟、跑负载测试、比较多个方案。
- Compound engineering 的价值在于把每次 agent 协作的经验反哺到未来指令中，让小的流程改进产生复利。

### 与当前 wiki 的连接

这篇文章和 [[hoard-things-you-know-how-to-do]] 是一组互补材料：前者强调囤积可运行例子作为 agent 输入资产，这篇强调把 agent 用于质量改进和技术债治理。它们共同指向一个更完整的 agentic engineering 工作流：先积累可复用 proof，再用 agent 低成本地组合、验证、重构和清理。

它也补充了已有的 `agent-engineer-quality` 方向：优秀的 agent 工程师不只是让 agent 写更多代码，而是设计出让代码库质量随时间上升的循环。质量不是交付速度的对立面；当重构和实验成本下降后，质量投资本身也可以成为日常流程的一部分。

### 个人判断

这篇文章最适合转化成一个工作习惯：遇到“早就该清理但没时间清理”的技术债，不要默认搁置，而是先判断它是否属于 agent 适合的后台任务。如果是，就开 worktree 或远程 agent，让它产出 PR，再人工评审。

对你的知识库来说，值得跟踪的不是“AI 可以提高质量”这个口号，而是具体可执行的模式：哪些重构适合 agent，PR 评审如何把关，探索性原型如何设计验证指标，以及每次任务结束后哪些经验应该进入 compound engineering loop。
